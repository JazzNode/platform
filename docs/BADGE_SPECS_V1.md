本文件定義 JazzNode 第一階段自動化徽章（Badges）系統的邏輯與實作規劃。

## 1. 核心設計原則
- **自動化優先**：基於 Airtable 現有資料進行計算。
- **可回溯性**：每個徽章需有明確的邏輯定義。
- **三對象覆蓋**：針對 Artists (樂手), Venues (場地), Users (會員) 三類對象設計。
- **現狀適配**：在「無 Check-in 資料」的前提下，以演出紀錄與追蹤數據為核心。

## 3. `badge_calculator.js` 流程規劃

### 3.1 資料抓取策略 (Efficiency)
為避免觸發 Airtable API 每秒 5 次的限制 (Rate Limit)，採取以下優化：
1. **批次處理**：使用 `select` 搭配 `formula` 僅抓取符合初步條件的 Records（例如：有 event 的 Artist 才抓）。
2. **快取統計**：在計算複雜指標（如 Top 10%）時，先一次性抓取所有目標對象的計數值，在記憶體中排序計算門檻，再進行分配。
3. **增量更新**：記錄上次計算時間，僅針對有異動（`updated_at`）的對象重算。

### 3.2 欄位更新機制
根據 `PROJECT_SCHEMA.md`，`badge_list` 欄位為 `multipleRecordLinks -> Badges`。
- **更新流程**：
  1. 腳本計算出該對象應擁有的 `badge_id` 列表。
  2. 查詢 `Badges` 表獲取對應的 `recordId`。
  3. 使用 Airtable API `update` 該對象的 `badge_list` 欄位，傳入 Record ID 陣列。
  4. **注意**：系統應保留「人工手動頒發」的徽章。計算邏輯應採用「合併」策略，除非該徽章屬於「系統自動撤回型」（如：近期不活躍而失去 Gig Warrior）。

### 3.3 定時任務 (Scheduler)
- 建議頻率：每日凌晨 (Daily at 03:00)。
- 針對 Artist 的演出頻率（過去 30/90 天）需要定期重算以反映即時狀態。

## 4. 未來擴展性
- **Check-in 整合**：未來若有打卡功能，可新增 `Jazz Pilgrim` (造訪過 N 個不同場地) 等徽章。
- **Ticket 整合**：基於 `Tickets_Orders` 實績頒發 `Patron` 徽章。
- **AI 語意分析**：從 `bio` 或 `description` 中提取隱藏特徵（如：專攻 Bebop 的樂手）。
