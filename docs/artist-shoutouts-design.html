🎤 藝人推薦系統設計

以場地留言系統為基礎，打造「推薦牆」— 整合社交證明、藝人策展、徽章激勵

一、核心概念

你的直覺完全正確 — 複用場地留言架構 + 重新定位

場地留言的完整架構（Provider → Form → List → Tags → Roles → Notifications → Dashboard）都可以複用。關鍵是改變定位和語氣：從「評價 / Review」變成「推薦 / Shoutout」，讓整個區塊的氛圍是正面的、鼓勵的。

命名方案

語言區塊標題按鈕文字表單提示
English Shoutouts Leave a Shoutout "Say something nice about [Artist]..."
繁體中文 來自音樂圈的推薦 留下推薦 「分享你對 [藝人名] 的推薦...」
日本語 ミュージシャンの声 推薦を書く 「[アーティスト名] への推薦を…」

💡 為什麼選「Shoutouts」而不是「Reviews」或「Recommendations」？
「Shoutout」在音樂圈是自然用語，意思是「公開的讚美/致敬」。它暗示正面、簡短、自發的推薦，不像 Review 可能帶有負面評價，也不像 Recommendation 那樣正式。中文用「來自音樂圈的推薦」則強調社群歸屬感。

二、場地留言 vs 藝人推薦 — 差異對照

🏠 場地留言（現有）

  • 任何人可留言，可正面可負面
  • Tags 是場地屬性（音響、飲料、交通...）
  • 場地無法「置頂」留言
  • 所有留言平等顯示
  • 有回覆功能
  • 可匿名

🎤 藝人推薦（新設計）

  • 任何人可推薦，語氣引導為正面
  • Tags 是藝人特質（音樂性、現場、合作、教學...）
  • 藝人可「📌 置頂」最多 3 則推薦
  • 藝人/場地推薦優先排序
  • 有回覆功能（藝人可回覆感謝）
  • 可匿名（但推薦通常不匿名）

新增的功能差異

功能場地留言藝人推薦
置頂 / Pin❌✅ 藝人可置頂最多 3 則
排序邏輯純時間倒序📌 Pinned → 🎵 藝人/場地 → 👤 一般 → 時間倒序
表單語氣中性（「Leave a comment」）正面引導（「Say something nice...」）
Tags場地屬性 8 個藝人特質 8 個（見下方）
sender_rolemember / venue_manager / artist / admin相同架構，完全複用
藝人頁面顯示—Pinned 顯示在「What People Say」區
全部顯示在底部「Shoutouts」區

五、置頂（Pin）機制

規則說明
誰可以 Pin只有 Claimed 藝人本人（tier ≥ 1）
最多幾則3 則（足夠顯示在「What People Say」區）
Pin 的位置「What People Say」精選區 + Shoutouts 列表最頂部
Pin UIDashboard 的推薦管理頁，每則推薦旁有 📌 按鈕
取消 Pin隨時可取消，推薦回到一般排序
是否 tier-gate建議所有 Claimed 都可以 Pin — Pin 是鼓勵大家經營頁面的工具

💡 Pin 是「藝人頁面即 EPK」策略的關鍵。藝人可以策展自己的推薦，挑選最有代表性的展示在頁面最顯眼位置。這等於給了他們自由編輯 EPK 「推薦語」區塊的能力，不需要我們做任何自訂排版。

六、徽章激勵機制

給留推薦的人（User Badges）

📣
First Shoutout
留下了第一則藝人推薦
條件：留下 1 則推薦
📢
Scene Supporter
積極推薦音樂圈的藝人
條件：給 5 位不同藝人留下推薦
🌟
Tastemaker
你的推薦被藝人置頂了！
條件：有 1 則推薦被藝人 Pin

給被推薦的藝人（Artist Badges）

❤️
Crowd Favorite
受到大家的喜愛與推薦
條件：收到 5+ 則推薦
🤝
Peer Respected
受到其他藝人的認可與推薦
條件：收到 3+ 則來自其他藝人的推薦
🏠
Venue's Choice
深受場地喜愛與推薦
條件：收到 3+ 家不同場地的推薦

💡 「Tastemaker」徽章是最聰明的設計。當使用者的推薦被藝人 Pin 時，使用者會收到通知 + 獲得徽章。這個雙重獎勵會大幅鼓勵人們留下高品質推薦，因為他們知道寫得好可能會被置頂。

觸發通知設計

事件通知對象通知內容
有人留了推薦被推薦的藝人「David Lin left a shoutout on your profile 🎤」
藝人 Pin 了推薦推薦的作者「Sarah Chen pinned your shoutout! 📌 You earned the Tastemaker badge 🌟」
藝人回覆推薦推薦的作者「Sarah Chen replied to your shoutout 💬」
獲得徽章獲得者「You earned the Scene Supporter badge! 📢」

七、資料庫設計

新表：artist_shoutouts

CREATE TABLE artist_shoutouts (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  artist_id    text NOT NULL,         -- 被推薦的藝人
  user_id      uuid NOT NULL REFERENCES profiles(id),

  -- 內容
  text         varchar(500),              -- 推薦文字（同 venue_comments 上限）
  tags         text[] DEFAULT '{}',    -- 特質標籤
  image_url    text,                      -- 可選附圖

  -- 身份（複用場地留言的 sender_role 模式）
  sender_role      text,                  -- null=member, 'artist', 'venue_manager', 'admin'
  sender_artist_id text,                  -- 如果以藝人身份留推薦
  sender_venue_id  text,                  -- 如果以場地身份留推薦（NEW）

  -- 置頂
  is_pinned    boolean DEFAULT false,
  pin_order    smallint,                  -- 1, 2, 3

  -- 選項
  is_anonymous boolean DEFAULT false,

  created_at   timestamptz DEFAULT now(),
  updated_at   timestamptz DEFAULT now()
);

-- 索引
CREATE INDEX idx_artist_shoutouts_artist ON artist_shoutouts(artist_id);
CREATE INDEX idx_artist_shoutouts_user   ON artist_shoutouts(user_id);

-- 回覆表（完全複用 venue_comment_replies 結構）
CREATE TABLE artist_shoutout_replies (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  shoutout_id  uuid NOT NULL REFERENCES artist_shoutouts(id) ON DELETE CASCADE,
  user_id      uuid NOT NULL REFERENCES profiles(id),
  sender_role  text,
  body         varchar(500),
  created_at   timestamptz DEFAULT now()
);

💡 為什麼不合併成一個通用 comments 表？
1. 場地留言已上線穩定運行，改動有風險
2. 藝人推薦有獨有欄位（is_pinned、pin_order、sender_venue_id）
3. 語氣和 tags 完全不同
4. RLS 規則不同
但前端元件可以高度複用 — Provider / Form / List 的架構相同，只是 props 和 config 不同。

九、鼓勵留推薦的設計手法

除了徽章，還有幾個 UX 手法可以鼓勵更多人留推薦：

1

「演出後提醒」推送

藝人演出結束 24 小時後，向有追蹤該藝人且參加了該場演出的粉絲發送通知：
「你昨晚在 Blue Note 看了 Sarah Chen 的演出，想留個推薦嗎？ 🎤」

DB 支援：lineups + follows + events.start_at 就可以算出來

2

「空推薦牆」引導

如果藝人頁面還沒有推薦，顯示空狀態引導：
「Be the first to leave a shoutout for Sarah Chen ✨」
搭配一個大的 CTA 按鈕，降低門檻。

3

「Tag-only 快速推薦」

允許只選 tags 不寫文字。有些人不善於文字表達，但願意點一下「🔥 Amazing Live」。
這些 tag-only 推薦不會出現在列表中，但會計入聚合統計（「12 people say: Amazing Live」）。

4

「合作者主動邀請」

在 Dashboard 提供「邀請合作者留推薦」功能：
藝人可以選擇常合作的人，系統發送一則友善的邀請通知。
這能快速從 0 建立起推薦基礎。

DB 支援：top collaborators 已從 lineups 計算好了

十、開發優先順序

Phase內容開發量

複用場地留言 70% code		 中（3-4 天）

Phase 3 聚合 Tags 統計 + 「What People Say」精選區 低（1-2 天）
Phase 4 徽章系統（6 個新 badges）+ Tag-only 快速推薦 低（1-2 天）

總計約 9-13 個工作天，如果場地留言系統穩定的話可以更快。Phase 1+2 上線後就能開始收集推薦了。

📋 總結

✅ 整合場地留言架構 — 70% 程式碼複用，大幅降低開發成本
✅ 「Shoutouts / 來自音樂圈的推薦」 — 正面語氣引導，不是 Review
✅ 藝人策展（Pin） — 最多 3 則，展示在頁面精華位置
✅ 排序優先 — 藝人/場地推薦自動排前面
✅ 聚合 Tags — 變成 LinkedIn 式技能背書統計

✅ Tag-only 快速推薦 — 降低門檻，累積數據
